# nftgeo-qos: Architecture & Design

This document outlines the architectural plan for introducing Traffic Control (QoS / Traffic Shaping) capabilities to the `nftgeo` firewall manager.

## 1. Overview and Motivation

`nftgeo` currently excels at packet filtering and geographic blocking using `nftables`. However, `nftables` by itself lacks the ability to queue, prioritize, and shape traffic (e.g., ensuring VoIP packets are never delayed by bulk HTTP downloads).

To achieve Next-Generation Firewall (NGFW) or Enterprise Router capabilities, we must integrate with the Linux Traffic Control (`tc`) subsystem. To maintain the stability of the core firewall script (`bin/nftgeo-update`), QoS logic will be separated into a distinct, dedicated component called `nftgeo-qos`.

## 2. Component Separation & Communication

### 2.1 The Two Engines
1.  **Firewall Engine (`bin/nftgeo-update`)**: Responsible for security, dropping malicious packets, and identifying traffic. When QoS is involved, this engine will merely attach a "mark" to the packet (e.g., `meta mark set 100`).
2.  **QoS Engine (`nftgeo-qos`)**: A new Go-based tool responsible for reading shaping requirements, creating `tc` queue disciplines (qdiscs) and classes, and directing marked packets into appropriate priority queues.

### 2.2 Single Source of Truth
State is not kept in a database. The configuration files in `/etc/nftgeo/` remain the Single Source of Truth.
- The UI writes draft files (e.g., `qos.conf` or extending `rules.conf`).
- The UI triggers an "Apply" action.
- The UI orchestrator invokes both `nftgeo-update --apply` (to load nft rules) and `nftgeo-qos apply` (to load tc rules).
- The existing "Deadman Switch" rollback mechanism will be extended so that a failure in either engine triggers a rollback for *both* firewall rules and QoS queues.

## 3. Addressing Edge Cases

### 3.1 Hierarchical QoS (VLANs and Bridges)
A naive QoS setup attempts to apply limits directly to `eth0`. If `eth0` has multiple VLANs (`eth0.10`, `eth0.20`), limiting `eth0` directly creates conflicts.
- **Solution:** The configuration format will define hierarchical relationships (e.g., defining `eth0` as the parent physical device with a global bandwidth ceiling, and VLANs as children).
- Under the hood, `nftgeo-qos` will utilize **IFB (Intermediate Functional Block)** devices. Traffic from multiple virtual interfaces can be redirected to a single `ifb` device where global shaping and classification safely occur before the packets hit the physical wire.

### 3.2 Orphaned Queues & Interface Flapping
Interfaces such as VPN tunnels (e.g., WireGuard `wg0`) or PPP links may drop and return dynamically. When a Linux interface disappears, its attached `tc` qdiscs are destroyed. When the interface returns, `tc` rules do not auto-reapply.
- **Solution:** While `nftgeo-qos` will act primarily as a CLI tool (`nftgeo-qos apply`), it will eventually need a daemon mode or deep integration with `systemd/udev` events. A Netlink listener will monitor for `RTM_NEWLINK` events. When an interface comes up, `nftgeo-qos` will automatically re-apply its specific queue disciplines without dropping the entire firewall state.

### 3.3 State Synchronization and Partial Failures
If `nftgeo-update` successfully applies rules but `nftgeo-qos` crashes (due to a typo in config or missing `tc` binary), packets will be marked but no queues will exist to handle them, potentially leading to default (or dropped) routing.
- **Solution:** Strict transactional application. The UI backend must monitor exit codes. If `tc` application fails, the UI must immediately revert the `nftables` rules to the previous working snapshot to guarantee system stability.

## 4. Host-based QoS (Local Services)

Because `nftgeo` is often deployed directly on application servers (rather than dedicated routers), it must handle locally generated traffic.

Using `nftables` output chains, the firewall can mark traffic generated by local processes. `nftgeo-qos` can then shape this traffic based on:
1.  **Source Port:** Prioritizing outgoing traffic from specific services (e.g., prioritizing `tcp 443` for the web server over `tcp 22` for SSH).
2.  **UID/GID (skuid):** Identifying and deprioritizing traffic generated by specific users (e.g., a background backup user).
3.  **Systemd Cgroups:** Modern Linux allows `nftables` to match packets directly to their originating Systemd service (`meta cgroup`). This allows guaranteeing bandwidth for `nginx.service` regardless of the ports it uses.

## 5. Implementation Roadmap

1.  **Phase 1 (Current):** Document architecture, update `nftgeo-update` to parse the `mark` keyword, and create the Go skeleton for `nftgeo-qos`.
2.  **Phase 2:** Implement `tc` shell command generation in Go. Support simple Root Qdisc (HTB) and classes on a single interface.
3.  **Phase 3:** Update the `nftgeo-ui` to support QoS configuration tabs and orchestrate the dual-apply workflow with rollback.
4.  **Phase 4:** Introduce IFB support for VLAN/Bridge hierarchies and daemon mode for dynamic interface monitoring.
